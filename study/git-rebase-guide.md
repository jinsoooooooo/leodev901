# 🛠 Git Rebase 해결 및 활용 가이드

`git pull` 과정에서 발생한 **Divergent Branches(갈라진 브랜치)** 문제를 해결하고 히스토리를 깔끔하게 정리하는 방법을 정리한 노트입니다.

## 1. 문제 상황 (Divergent Branches)
로컬과 원격 저장소가 서로 다른 커밋을 가지고 있어 Git이 어떻게 합칠지 모르는 상태입니다.
- **로컬**: `feat(projects): 프로젝트 이력 한글 업데이트` (나의 작업)
- **원격**: `feat: learning 실습 콘텐츠 추가` (타인의 작업 또는 다른 곳에서의 작업)

## 2. 해결 방법: `git pull --rebase`
히스토리를 일직선으로 유지하며 동기화하는 가장 깔끔한 방법입니다.

```bash
git pull --rebase origin main
```

### ⚙️ 동작 원리 (Rebase)
1.  **임시 저장**: 로컬의 새로운 커밋들을 잠시 따로 보관합니다.
2.  **동기화**: 원격(`origin/main`)의 최신 커밋들을 내 로컬로 가져옵니다.
3.  **재배치**: 따로 보관했던 로컬 커밋들을 원격 커밋들 **"위에"** 하나씩 다시 쌓습니다.

## 3. Rebase vs Merge: 언제 무엇을 쓸까?

| 구분 | Git Rebase | Git Merge |
| :--- | :--- | :--- |
| **특징** | 히스토리를 한 줄로 재작성함 | 두 히스토리를 합친 'Merge 커밋' 생성 |
| **장점** | 히스토리가 매우 깔끔함 (선형적) | 병합된 기록이 명확히 남음 |
| **언제?** | **로컬 개인 작업**을 동기화할 때 | **중요한 기능**을 메인에 합칠 때 |
| **주의** | 이미 Push된 커밋엔 사용 금지 | 히스토리가 거미줄처럼 복잡해질 수 있음 |

> [!IMPORTANT]
> **Golden Rule**: 이미 원격(GitHub 등)에 올린 커밋은 절대 `rebase`하지 마세요. 히스토리가 바뀌어 협업자들의 Git이 꼬일 수 있습니다.

## 4. 유용한 확인 명령어

```bash
# 1. 히스토리 그래프로 보기
git log --oneline --graph --all

# 2. 현재 상태 확인
git status

# 3. (Rebase 중 충돌 발생 시) 충돌 해결 후 진행
git add .
git rebase --continue
```

---
> 이 노트는 `git pull` 시 발생하는 분기 문제를 rebase 전략으로 해결한 사례를 바탕으로 작성되었습니다.
